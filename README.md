## 15. Задания 2 семестра ##

1. Создать репозиторий для проекта
Список назначенных проектов:
https://docs.google.com/spreadsheets/d/1MD0GeCpA8NQHagB2AJxmbIn8qx0CPGznbtV_eNQrRLc/edit?usp=sharing

2. Создать структуру папок в репозитории как в
https://github.com/vilegzhanina/ProgrammingCourse
(report, review, source, README.md, .gitignore)

Синтаксис маркдаун
https://help.github.com/categories/writing-on-github/

3. Добавить ссылку на свой репозиторий в
https://github.com/vilegzhanina/ProgrammingCourse/blob/master/list.md

4. Заполнить README.md по шаблону
https://github.com/vilegzhanina/ProgrammingCourse/blob/master/README.md

5. Установить UML редактор -- выбрать какой понравится
dia, astah, staruml, другое
Построить в нем свои диаграммы, исходники диаграмм сохранить в папке report,
экспортированные картинки -- туда же.

6. Задать архитектуру проекта
а) Построить диаграмму компонентов,
б) создать qt проекты под каждый компонент (библиотеку, приложение),
в) создать интерфейс класса (только header file с классом,
который будет предоставлять функции ядра системы,
документировать его в нотации doxygen.

7. Доходчиво попросить коллегу из списка сделать ревью
https://docs.google.com/spreadsheets/d/1MD0GeCpA8NQHagB2AJxmbIn8qx0CPGznbtV_eNQrRLc/edit?usp=sharing

Предварительно следует добавить его в collaborators,
Если Иван получив просьбу от Михаила делает ревью для Михаила, то Иван 
размещает в репозитории Михаила
в папке review файл 1.Arch_review, в котором перечисляет различные 
замечания,
которые Ивану удалось придумать.

Следует указывать все возможные замечания, начиная от орфографии, 
пунктуации и до замечаний по коду.
Важно обратить внимание на диаграммы и полноту описания задания.
Сейчас чем больше, тем лучше.
Старайтесь при ревью не навязывать свое видение задачи,
а оценить полноту и непротиворечивость изложенного видения.
Смогли бы вы по полученному описанию сделать софт, который бы в итоге 
отвечал мечтам автора описания?
То есть, достаточно ли сведений?
Если какие-то шаги из пунктов 1-6 не были выполнены, то это тоже следует 
отметить.

8. Получив результаты ревью следует либо согласится с каждым из пунктов 
по отдельности
и внести исправления в репозиторий,
либо прокомментировать, если не согласны.

9. Возможно повторить пункты 7-8.

## 23. Qt книга Шлее ##
книга Макса Шлее про Qt,
https://www.assembla.com/spaces/octocoder/documents/aotJTS9Her46C-acwqjQXA/download/aotJTS9Her46C-acwqjQXA
800+ страниц -- все как вы любите.
Большая половина из нее понадобится.

Книга про версию 4.8, но, думаю, нам достаточно.
Уже вышла книга этого же автора про 5.3,
но я сходу не нашла, где ее можно достать.
Если найдете, вы знаете что делать.

Наибольшим приоритетом для нас сейчас является выпуск mvp релиза,
который у всех нас не подразумевает gui, так что не следует бросаться в qt.

Но если вы все же не сможете удержаться, сделайте в своем репозитории ветку,
назовите ее типа gui_app, и в ней равзлекайтесь.
Если понадобятся методы из ядра, которые добавятся после ответвления,
вы всегда можете их подлить из основной ветки в gui_app.
Таким образом, не возбраняется делать gui, для функциональностей
уже реализованных методов ядра в отдельной ветке.
Но надо помнить про необходимость как-только-так-сразу выпуска mvp.
